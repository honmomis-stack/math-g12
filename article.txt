របាយការណ៍ស្រាវជ្រាវស្តីពីស្ថាបត្យកម្មប្រព័ន្ធ និងការរចនាបទពិសោធន៍អ្នកប្រើប្រាស់សម្រាប់កម្មវិធីគេហទំព័រសៀវភៅគណិតវិទ្យាឌីជីថល
ការវិវត្តនៃបច្ចេកវិទ្យាអប់រំ (EdTech) បានផ្លាស់ប្តូរទម្រង់នៃការសិក្សាពីសៀវភៅបោះពុម្ពបែបប្រពៃណី ទៅជាប្រព័ន្ធឌីជីថលដែលមានអន្តរកម្មខ្ពស់។ ការអភិវឌ្ឍកម្មវិធីគេហទំព័រ (Web Application) ជាទម្រង់សៀវភៅគណិតវិទ្យាឌីជីថល ដែលមានការបែងចែកជាជំពូក មេរៀនសង្ខេប រូបមន្ត លំហាត់គំរូ សំណួរពហុជ្រើសរើស (MCQ) និងលំហាត់អនុវត្តដែលមានប្រព័ន្ធជំនួយ (Hints) ទាមទារឱ្យមានការរចនាស្ថាបត្យកម្មប្រព័ន្ធយ៉ាងរឹងមាំ។ របាយការណ៍នេះធ្វើការវិភាគស៊ីជម្រៅទៅលើការរៀបចំមូលដ្ឋានទិន្នន័យ បច្ចេកវិទ្យាបង្ហាញរូបមន្តគណិតវិទ្យា (LaTeX) ឧបករណ៍គូរក្រាបអន្តរកម្ម តក្កវិជ្ជានៃការគ្រប់គ្រងលំហាត់ (State Logic) និងគោលការណ៍ចិត្តសាស្ត្រនៃការរចនាបទពិសោធន៍អ្នកប្រើប្រាស់ (UX/UI) ដើម្បីធានាបាននូវបរិស្ថានសិក្សាដ៏មានប្រសិទ្ធភាព។
ស្ថាបត្យកម្មមូលដ្ឋានទិន្នន័យ និងការធ្វើម៉ូដែលទិន្នន័យ (Database Architecture and Data Modeling)
ការកសាងកម្មវិធីអប់រំដែលផ្ទុកមាតិកាស្មុគស្មាញ តម្រូវឱ្យមានការរចនាមូលដ្ឋានទិន្នន័យតាមបែបទំនាក់ទំនង (Relational Database) ដូចជា PostgreSQL ជាដើម ដែលផ្តល់នូវភាពបត់បែន និងសមត្ថភាពក្នុងការគ្រប់គ្រងរចនាសម្ព័ន្ធទិន្នន័យត្រួតស៊ីគ្នាបានយ៉ាងល្អ1។ នៅក្នុងការរចនាប្រព័ន្ធនេះ គេត្រូវគិតគូរដល់ស្ថាបត្យកម្មបីកម្រិត (Three-Schema Architecture) ដែលរួមមានកម្រិតរូបវន្ត (Physical Schema) កម្រិតគំនិត (Conceptual Schema) និងកម្រិតខាងក្រៅ (External Schema ឬ View) ដើម្បីធានាបាននូវឯករាជ្យភាពនៃទិន្នន័យ និងសុវត្ថិភាព3។
ដើម្បីឆ្លើយតបទៅនឹងតម្រូវការនៃការបែងចែកសៀវភៅជាមេរៀន មាតិការង និងប្រភេទលំហាត់ផ្សេងៗ ការអនុវត្តគោលការណ៍ Normalization គឺជារឿងចាំបាច់បំផុតក្នុងការលុបបំបាត់ភាពជាន់គ្នានៃទិន្នន័យ និងធានាបាននូវបូរណភាពទិន្នន័យ (Data Integrity)3។ ការរៀបចំតារាងនីមួយៗត្រូវមានទំនាក់ទំនងយ៉ាងច្បាស់លាស់តាមរយៈកូនសោចម្បង (Primary Key) និងកូនសោរង (Foreign Key)1។

បច្ចេកវិទ្យាបង្ហាញរូបមន្តគណិតវិទ្យា៖ ការជ្រើសរើសនិងការអនុវត្ត (LaTeX Rendering)
បញ្ហាប្រឈមដ៏ធំបំផុតមួយក្នុងការអភិវឌ្ឍកម្មវិធីគេហទំព័រសម្រាប់មុខវិជ្ជាវិទ្យាសាស្ត្រពិត គឺភាសា HTML មិនត្រូវបានរចនាឡើងពីកំណើតដើម្បីគាំទ្រដល់ការបង្ហាញនិមិត្តសញ្ញាគណិតវិទ្យាស្មុគស្មាញ និងប្រភាគនោះទេ9។ ជាប្រវត្តិសាស្ត្រ អ្នកអភិវឌ្ឍន៍តែងតែព្យាយាមប្រើប្រាស់ស្លាក <span> ជាច្រើនរួមផ្សំជាមួយ CSS ដើម្បីតម្រឹមរូបមន្ត ដែលជាវិធីសាស្ត្រមិនមានប្រសិទ្ធភាព និងងាយនឹងបែកទ្រង់ទ្រាយនៅពេលចម្លង (Copy-pasting)9។ ថ្វីបើបច្ចុប្បន្នមានបច្ចេកវិទ្យា MathML ដែលត្រូវបានគាំទ្រដោយកម្មវិធីរុករក (Browsers) មួយចំនួនក៏ដោយ ក៏វានៅតែមានភាពខ្វះចន្លោះនៅក្នុងការគាំទ្រមុខងារពេញលេញបើប្រៀបធៀបទៅនឹងបណ្ណាល័យជំនាញ9។ ហេតុនេះ បណ្ណាល័យ JavaScript ភាគីទីបីដូចជា MathJax និង KaTeX ត្រូវបានគេចាត់ទុកជាស្តង់ដារមាសសម្រាប់ការបង្ហាញកូដ LaTeX នៅលើគេហទំព័រ9។

ផ្ទុយទៅវិញ MathLive គឺជាបណ្ណាល័យជំនាន់ថ្មីដែលត្រូវបានរចនាឡើងយ៉ាងពិសេសសម្រាប់កម្មវិធីគេហទំព័រទំនើបៗ។ វាប្រើប្រាស់បច្ចេកវិទ្យា Web Components (<math-field>) ដែលអាចធ្វើសមាហរណកម្មជាមួយ React បានយ៉ាងងាយស្រួល31។ អត្ថប្រយោជន៍ដ៏ធំបំផុតរបស់ MathLive គឺការផ្តល់នូវក្តារចុចនិម្មិត (Virtual Keyboards) ដែលមានរូបមន្តស្រាប់ ធ្វើឱ្យសិស្សងាយស្រួលក្នុងការវាយបញ្ចូលរូបមន្តស្មុគស្មាញពីឧបករណ៍ចល័ត (Smartphones) ព្រមទាំងគាំទ្រដល់អ្នកប្រើប្រាស់ដែលពិការភ្នែកតាមរយៈបច្ចេកវិទ្យាអានអេក្រង់ (Screen-reader friendly)31។ លើសពីនេះ MathLive អាចបញ្ចេញលទ្ធផលមិនត្រឹមតែជាកូដ LaTeX ប៉ុណ្ណោះទេ ថែមទាំងជាទម្រង់ Abstract Syntax Tree (AST) ដែលអនុញ្ញាតឱ្យប្រព័ន្ធតក្កវិជ្ជាអាចផ្ទៀងផ្ទាត់ភាពត្រឹមត្រូវនៃចម្លើយតាមលក្ខណៈគណិតវិទ្យាពិតប្រាកដ (ឧទាហរណ៍ ប្រព័ន្ធអាចយល់ថា  និង  គឺជាចម្លើយតែមួយ)31។
តក្កវិជ្ជាគ្រប់គ្រងសកម្មភាពនៅក្នុង React (State Management)
ការគ្រប់គ្រងវឌ្ឍនភាពនៃការធ្វើលំហាត់ តម្រូវឱ្យមានការគ្រប់គ្រងស្ថានភាព (State Management) យ៉ាងប្រុងប្រយ័ត្ន។ ទោះបីជា useState របស់ React អាចត្រូវបានប្រើប្រាស់សម្រាប់រក្សាទុកព័ត៌មានសាមញ្ញក៏ដោយ ប៉ុន្តែសម្រាប់លំហាត់ដែលមានភាពស្មុគស្មាញ ដូចជាការតាមដានសំណួរបច្ចុប្បន្ន ការកត់ត្រាចម្លើយរបស់អ្នកប្រើប្រាស់ ការគណនាពិន្ទុ និងការកត់ត្រាចំនួនដងនៃការប្រើប្រាស់ជំនួយ (Hints) នោះការប្រើប្រាស់ useReducer គឺជាស្តង់ដារវិស្វកម្មដែលត្រឹមត្រូវបំផុត35។
ការរចនាតាមបែប useReducer អនុញ្ញាតឱ្យផ្តុំតក្កវិជ្ជាអាជីវកម្ម (Business Logic) ទាំងអស់ទៅក្នុងកន្លែងតែមួយ ដែលធ្វើឱ្យកូដមានភាពងាយស្រួលក្នុងការធ្វើតេស្ត និងថែទាំ36។ សកម្មភាព (Actions) សំខាន់ៗដែលកម្មវិធីត្រូវបញ្ជូន (Dispatch) រួមមាន៖
	•	SUBMIT_ANSWER: នៅពេលសិស្សចុចបញ្ជូនចម្លើយ ប្រព័ន្ធនឹងយកទិន្នន័យ (Payload) ទៅប្រៀបធៀបជាមួយចម្លើយត្រឹមត្រូវនៅក្នុងមូលដ្ឋានទិន្នន័យ។ ប្រសិនបើទម្រង់ MCQ វាគ្រាន់តែផ្ទៀងផ្ទាត់ ID ប៉ុន្តែបើជាទម្រង់វាយបញ្ចូល វានឹងដំណើរការការផ្ទៀងផ្ទាត់ AST ដែលបានលើកឡើងខាងលើ6។
	•	NEXT_QUESTION: សម្រាប់បង្កើនលេខរៀងសំណួរបន្តបន្ទាប់។
	•	SHOW_HINT: សម្រាប់កត់ត្រាសកម្មភាពនៅពេលសិស្សត្រូវការជំនួយ ដែលអាចកាត់ពិន្ទុបន្តិចបន្តួចដើម្បីលើកទឹកចិត្តឱ្យពួកគេព្យាយាមគិតដោយខ្លួនឯងជាមុន។
	•	RESET_QUIZ: សម្រាប់ត្រឡប់ស្ថានភាពទាំងអស់ទៅកាន់ចំណុចចាប់ផ្តើម36។
គោលការណ៍នៃការរចនាបទពិសោធន៍អ្នកប្រើប្រាស់ និងចំណុចប្រទាក់ (UX/UI Principles for EdTech)
ការរចនាកម្មវិធីអប់រំមិនមែនគ្រាន់តែផ្តោតលើភាពស្រស់ស្អាតនៃចំណុចប្រទាក់ប៉ុណ្ណោះទេ ប៉ុន្តែត្រូវពឹងផ្អែកយ៉ាងខ្លាំងទៅលើចិត្តសាស្ត្រនៃការរៀនសូត្រ (Cognitive Psychology)37។ ប្រព័ន្ធឌីជីថលដែលរចនាខុស អាចធ្វើឱ្យសិស្សមានភាពធុញថប់ និងបោះបង់ការសិក្សាយ៉ាងឆាប់រហ័ស។
ការវាយតម្លៃលើកម្មវិធីសិក្សាល្បីៗដូចជា Khan Academy និង Brilliant បានបង្ហាញពីទស្សនវិជ្ជាពីរខុសគ្នា។ ខណៈដែល Khan Academy ផ្តោតលើការរៀនដោយការមើលវីដេអូនិងអនុវត្តតាមក្រោយ (Learn by watching) កម្មវិធី Brilliant បានជ្រើសរើសផ្លូវនៃការរៀនសូត្របែបសកម្ម (Active Learning) ដោយអនុញ្ញាតឱ្យសិស្សធ្វើអន្តរកម្មជាមួយនឹងបញ្ហាផ្ទាល់តាំងពីជំហានដំបូង (Learn by doing)39។ សម្រាប់កម្មវិធីសៀវភៅគណិតវិទ្យានេះ ការយកគំរូតាមវិធីសាស្ត្រសកម្មរបស់ Brilliant នឹងជួយកសាងការយល់ដឹងស៊ីជម្រៅ (Conceptual Understanding) ដល់សិស្ស39។
ដើម្បីសម្រេចបាននូវគោលដៅនេះ ការរចនា UX/UI ត្រូវអនុវត្តតាមគោលការណ៍សំខាន់ៗដូចខាងក្រោម៖
	•	ការកាត់បន្ថយបន្ទុកការគិត (Reducing Cognitive Load) និង ច្បាប់ Hick's Law: យោងតាមច្បាប់ Hick's Law ការផ្តល់ជម្រើសកាន់តែច្រើនដល់អ្នកប្រើប្រាស់ នឹងធ្វើឱ្យពួកគេចំណាយពេលសម្រេចចិត្តកាន់តែយូរ និងងាយនឹងមានអារម្មណ៍ស្មុគស្មាញ37។ ដូច្នេះ ចំណុចប្រទាក់នៃមេរៀននីមួយៗត្រូវរចនាឡើងយ៉ាងសាមញ្ញ (Minimalist) ដោយបែងចែកមាតិកាជាផ្នែកតូចៗ (Chunking)។ ម៉ឺនុយរុករកគួរតែមានជម្រើសមិនលើសពី 5 ទៅ 7 នោះទេ37។
	•	ការបង្ហាញបន្តិចម្តងៗ (Progressive Disclosure): ដើម្បីចៀសវាងការផ្ទុកព័ត៌មានលើសទម្ងន់ (Information Overload) ព័ត៌មានលម្អិតដូចជាការពន្យល់ស៊ីជម្រៅអំពីទ្រឹស្តីបទ គួរតែត្រូវបានលាក់ទុកជាបណ្ដោះអាសន្ន និងបង្ហាញតែនៅពេលដែលសិស្សចុចប៊ូតុង "អានបន្ថែម" ឬតាមរយៈ Modal Windows ប៉ុណ្ណោះ37។
	•	រចនាសម្ព័ន្ធនៃការអាន (F-Shape Pattern): ការសិក្សាពីវិទ្យាស្ថាន Nielsen Norman Group បង្ហាញថា មនុស្សភាគច្រើនអានគេហទំព័រជារាងអក្សរ F41។ ដូច្នេះ ព័ត៌មានដែលសំខាន់បំផុត ដូចជាខ្លឹមសារសំណួរ ឬរូបមន្តគន្លឹះ គួរតែត្រូវបានដាក់នៅផ្នែកខាងឆ្វេងផ្នែកខាងលើនៃទំព័រ រីឯប៊ូតុងអន្តរកម្ម (Call-to-Action) គួរដាក់នៅទីតាំងដែលងាយស្រួលចុចនៅផ្នែកខាងក្រោម41។
	•	ការផ្តល់មតិកែលម្អភ្លាមៗ (Immediate Visual Feedback): ការរៀនសូត្រមានប្រសិទ្ធភាពបំផុតនៅពេលមានការឆ្លើយតបភ្លាមៗ។ នៅពេលសិស្សឆ្លើយសំណួរ ប្រព័ន្ធត្រូវប្រើប្រាស់ពណ៌ឱ្យបានច្បាស់លាស់ (ពណ៌បៃតងតំណាងឱ្យភាពត្រឹមត្រូវ ពណ៌ក្រហមតំណាងឱ្យកំហុស) រួមជាមួយនឹងការពន្យល់ខ្លីៗពីមូលហេតុនៃកំហុសនោះ38។ ការផ្តល់របារវឌ្ឍនភាព (Progress Bars) ដែលបង្ហាញពីចំនួនលំហាត់ដែលបានបញ្ចប់ ក៏ជាកត្តាជំរុញទឹកចិត្តយ៉ាងសំខាន់ផងដែរ37។
យន្តការផ្តល់ជំនួយបណ្តើរៗ និងការជំរុញការគិត (Progressive-Hint Prompting System)
លក្ខណៈពិសេសចម្បងមួយនៃកម្មវិធីនេះ គឺការផ្តល់នូវ "Tips ខ្លីៗ" នៅក្នុងលំហាត់អនុវត្ត។ ទោះជាយ៉ាងណាក៏ដោយ ការស្រាវជ្រាវផ្នែកអប់រំបានបង្ហាញថា ការផ្តល់ជំនួយពេញលេញភ្លាមៗនៅពេលសិស្សទាមទារ (On-demand hints) ជារឿយៗមិននាំមកនូវលទ្ធផលល្អនោះទេ ពីព្រោះសិស្សអាចនឹងគ្រាន់តែយកចម្លើយដើម្បីបញ្ចៀសការលំបាក ដោយមិនបានស្វែងយល់ពីតក្កវិជ្ជានៃការដោះស្រាយ ឬហៅថាការទាយជាប្រព័ន្ធ (Systematic guessing)42។
ដើម្បីដោះស្រាយបញ្ហានេះ កម្មវិធីត្រូវអនុវត្តយន្តការ Progressive-Hint Prompting (PHP)។ វិធីសាស្ត្រនេះស្រដៀងទៅនឹងបច្ចេកទេស Chain-of-Thought ដែលប្រើប្រាស់នៅក្នុងការបង្ហាត់បញ្ញាសិប្បនិម្មិត (AI) ដែរ ដែលវាមានគោលដៅជំរុញការគិតជាជំហានៗ43។ ប្រព័ន្ធ PHP ដំណើរការនៅក្នុងកម្មវិធីដោយមានទម្រង់ជាជំហានៗ ដូចខាងក្រោម៖
	•	ជំនួយកម្រិតទី 1 (Conceptual Hint): នៅពេលសិស្សចុចសុំជំនួយលើកដំបូង ប្រព័ន្ធមិនត្រូវបង្ហាញតួលេខ ឬការគណនានោះទេ ប៉ុន្តែត្រូវរំលឹកពីទ្រឹស្តីបទ ឬរូបមន្តគោលដែលពាក់ព័ន្ធនឹងលំហាត់នោះ (ឧទាហរណ៍៖ "សូមគិតអំពីទ្រឹស្តីបទពីតាហ្គ័រ ")42។
	•	ជំនួយកម្រិតទី 2 (Procedural Hint): ប្រសិនបើសិស្សនៅតែមិនទាន់អាចដោះស្រាយបាន ជំនួយលើកទីពីរនឹងបង្ហាញពីជំហានដំបូងនៃការគណនា ឬការជំនួសលេខចូលទៅក្នុងរូបមន្ត ដើម្បីផ្តល់ចំណុចចាប់ផ្តើមដ៏រឹងមាំមួយ43។
	•	ជំនួយកម្រិតទី 3 (Solution Proximity Hint): ជាជំនួយចុងក្រោយ ដែលបង្ហាញការដោះស្រាយស្ទើរតែដល់ទីបញ្ចប់ ដោយបន្សល់ទុកតែប្រតិបត្តិការគណនាចុងក្រោយប៉ុណ្ណោះ ដើម្បីឱ្យសិស្សនៅតែជាអ្នករកឃើញចម្លើយចុងក្រោយដោយខ្លួនឯង43។
ការរចនាប្រព័ន្ធជំនួយបែបនេះ ធានាថាសិស្សមិនក្លាយជាអ្នកទទួលយកចម្លើយដោយអកម្មឡើយ ប៉ុន្តែត្រូវបានដឹកនាំ និងណែនាំយ៉ាងយកចិត្តទុកដាក់ប្រៀបដូចជាមានគ្រូបង្រៀនផ្ទាល់ខ្លួននៅក្បែរ ដែលជួយបណ្តុះបណ្តាលពួកគេឱ្យចេះដោះស្រាយបញ្ហាប្រកបដោយភាពជឿជាក់និងឯករាជ្យភាព។
សេចក្តីសន្និដ្ឋាន
ការបង្កើតកម្មវិធីគេហទំព័រសៀវភៅគណិតវិទ្យាឌីជីថលដែលប្រកបដោយអន្តរកម្ម និងប្រសិទ្ធភាពខ្ពស់ គឺជាគម្រោងដ៏ស្មុគស្មាញមួយ ដែលទាមទារឱ្យមានការសហការគ្នាយ៉ាងជិតស្និទ្ធរវាងស្ថាបត្យកម្មវិស្វកម្មផ្នែកទន់ និងគោលការណ៍គរុកោសល្យ។
ការកសាងគ្រឹះដ៏រឹងមាំចាប់ផ្តើមពីការរចនាមូលដ្ឋានទិន្នន័យ (Relational Database) ដែលមានទំនាក់ទំនងច្បាស់លាស់រវាងមេរៀន សំណួរ និងចម្លើយ ដោយគោរពតាមស្ថាបត្យកម្ម Three-Schema។ សម្រាប់ផ្នែកបង្ហាញ (Front-end) ការប្រើប្រាស់បណ្ណាល័យ KaTeX ឆ្លងកាត់ប្រព័ន្ធ Markdown (react-markdown) ធានាបាននូវការបង្ហាញរូបមន្ត LaTeX ប្រកបដោយល្បឿនលឿន មិនមានបញ្ហាលោតទំព័រ និងមានសុវត្ថិភាពខ្ពស់ពីការវាយប្រហារតាមអ៊ីនធឺណិត។ ចំណែកឯការធ្វើអន្តរកម្មលើក្រាប ការដាក់បញ្ចូលគ្នារវាង Function-Plot សម្រាប់ការគូរក្រាបសាមញ្ញ និង JSXGraph សម្រាប់ធរណីមាត្រស្មុគស្មាញ នឹងធ្វើឱ្យមេរៀនមានជីវិតរស់រវើក។
លើសពីនេះ ការប្រើប្រាស់ MathLive សម្រាប់ការវាយបញ្ចូលរូបមន្តគណិតវិទ្យា និងការគ្រប់គ្រងតក្កវិជ្ជាលំហាត់តាមរយៈ useReducer ផ្តល់នូវបទពិសោធន៍ដ៏រលូនសម្រាប់ការធ្វើតេស្តសមត្ថភាពសិស្ស។ ជាចុងក្រោយ ភាពជោគជ័យដ៏ពិតប្រាកដនៃកម្មវិធីនេះ គឺអាស្រ័យទៅលើការអនុវត្តគោលការណ៍ UX/UI យ៉ាងម៉ត់ចត់ ពិសេសគឺការកាត់បន្ថយបន្ទុកការគិត (Cognitive Load) និងការប្រើប្រាស់ប្រព័ន្ធជំនួយបែប PHP (Progressive-Hint Prompting) ដែលជួយបំប្លែងដំណើរការសិក្សាពីភាពតានតឹង ទៅជាបទពិសោធន៍មួយដែលពោរពេញទៅដោយការលើកទឹកចិត្ត និងភាពជោគជ័យ។ ដោយប្រកាន់ខ្ជាប់នូវគោលការណ៍និងបច្ចេកវិទ្យាទាំងនេះ កម្មវិធីសៀវភៅឌីជីថលនេះនឹងក្លាយជាឧបករណ៍អប់រំដ៏មានឥទ្ធិពល ដែលឆ្លើយតបយ៉ាងពេញលេញទៅនឹងទម្រង់នៃការសិក្សាក្នុងយុគសម័យទំនើប។
